# Requirements: Player Stats System

## Overview

Система характеристик (stats) для BoundByFate-Core реализует D&D 5e-подобные атрибуты для игроков и мобов. Система должна быть гибкой, расширяемой и интегрироваться с механиками Minecraft.

## Functional Requirements

### FR-1: Core Stats Management

**FR-1.1: Six Base Stats**
- Система ДОЛЖНА поддерживать 6 базовых характеристик D&D 5e:
  - Сила (STR) - `boundbyfate-core:strength`
  - Выносливость (CON) - `boundbyfate-core:constitution`
  - Ловкость (DEX) - `boundbyfate-core:dexterity`
  - Интеллект (INT) - `boundbyfate-core:intelligence`
  - Мудрость (WIS) - `boundbyfate-core:wisdom`
  - Харизма (CHA) - `boundbyfate-core:charisma`

**FR-1.2: D&D Modifier Calculation**
- Система ДОЛЖНА вычислять D&D модификатор по формуле: `modifier = floor((value - 10) / 2)`
- Модификатор ДОЛЖЕН пересчитываться автоматически при изменении значения характеристики

**FR-1.3: Value Constraints**
- Каждая характеристика ДОЛЖНА иметь минимальное значение (по умолчанию 1)
- Каждая характеристика ДОЛЖНА иметь максимальное значение (по умолчанию 30)
- Значения ДОЛЖНЫ автоматически зажиматься в допустимый диапазон

### FR-2: Extensible Stat System

**FR-2.1: Custom Stats Registration**
- Система ДОЛЖНА позволять регистрировать новые характеристики через API
- Новые характеристики ДОЛЖНЫ иметь уникальный Identifier
- Регистрация ДОЛЖНА происходить до загрузки конфигов

**FR-2.2: Stat Definition Properties**
- Каждая характеристика ДОЛЖНА иметь:
  - Уникальный ID (Identifier)
  - Короткое имя (например "STR")
  - Отображаемое имя (например "Сила")
  - Минимальное значение
  - Максимальное значение
  - Значение по умолчанию

### FR-3: Modifier System

**FR-3.1: Modifier Sources**
- Система ДОЛЖНА поддерживать модификаторы из разных источников:
  - Расовые бонусы (race)
  - Классовые бонусы (class)
  - Бонусы от уровня (level)
  - Временные эффекты (effects)
  - Админские изменения (admin)

**FR-3.2: Modifier Types**
- Система ДОЛЖНА поддерживать типы модификаторов:
  - FLAT - добавляет значение к итоговому результату
  - OVERRIDE - заменяет базовое значение (для админских команд)

**FR-3.3: Modifier Stack Management**
- Модификаторы ДОЛЖНЫ складываться аддитивно
- Система ДОЛЖНА позволять удалять все модификаторы от конкретного источника
- Модификаторы ДОЛЖНЫ сохраняться вместе с данными сущности

### FR-4: Entity Stats Attachment

**FR-4.1: Player Stats**
- Каждый игрок ДОЛЖЕН иметь прикреплённые данные характеристик (EntityStatData)
- Данные ДОЛЖНЫ загружаться из конфига при входе на сервер
- Данные ДОЛЖНЫ сохраняться в NBT файлах мира

**FR-4.2: Mob Stats**
- Мобы ДОЛЖНЫ иметь возможность иметь характеристики
- Характеристики мобов ДОЛЖНЫ загружаться из конфигов по типу моба
- Мобы без конфига ДОЛЖНЫ получать дефолтные значения

**FR-4.3: Data Persistence**
- EntityStatData ДОЛЖЕН использовать Fabric Data Attachment API
- Сериализация ДОЛЖНА использовать Codec (как в PlayerLevelData)
- Данные ДОЛЖНЫ переживать смерть игрока
- Данные ДОЛЖНЫ переживать перезагрузку мира

### FR-5: Configuration System

**FR-5.1: Character Configs**
- Конфиги персонажей ДОЛЖНЫ храниться в `world/boundbyfate/characters/{playerName}.json`
- Формат конфига ДОЛЖЕН включать:
  - Имя игрока (playerName)
  - Расу (race)
  - Класс (characterClass)
  - Стартовый уровень (startingLevel)
  - Базовые значения характеристик (baseStats)

**FR-5.2: Mob Configs**
- Конфиги мобов ДОЛЖНЫ храниться в `world/boundbyfate/mobs/{mobType}.json`
- Формат конфига ДОЛЖЕН включать:
  - Тип моба (mobTypeId)
  - Базовые значения характеристик (baseStats)

**FR-5.3: Config Validation**
- Система ДОЛЖНА валидировать значения характеристик при загрузке
- Невалидные значения ДОЛЖНЫ зажиматься в допустимый диапазон
- Ошибки валидации ДОЛЖНЫ логироваться с указанием файла и поля
- Невалидные конфиги НЕ ДОЛЖНЫ крашить сервер

**FR-5.4: Config Reload**
- Система ДОЛЖНА поддерживать перезагрузку конфигов без перезапуска сервера
- Команда `/bbf stats reload` ДОЛЖНА перезагружать все конфиги
- При reload ДОЛЖНЫ переприменяться эффекты к существующим сущностям

### FR-6: Minecraft Integration (Effects)

**FR-6.1: Effect System Architecture**
- Система ДОЛЖНА иметь расширяемый слой эффектов (StatEffect)
- Каждая характеристика МОЖЕТ иметь привязанные эффекты (StatEffectBinding)
- Эффекты ДОЛЖНЫ применяться автоматически при изменении характеристик

**FR-6.2: Built-in Effects**
- Система ДОЛЖНА предоставлять встроенные эффекты:
  - EntityAttributeEffect - изменение EntityAttribute
  - MaxHealthEffect - изменение максимального HP
  - MovementSpeedEffect - изменение скорости передвижения
  - StatusEffectStatEffect - применение StatusEffect

**FR-6.3: Effect Application**
- Эффекты ДОЛЖНЫ применяться при:
  - Входе игрока на сервер
  - Изменении модификаторов
  - Ручном изменении через команды
  - Reload конфигов
- Старые эффекты ДОЛЖНЫ удаляться перед применением новых (идемпотентность)

**FR-6.4: Custom Effects**
- Другие моды ДОЛЖНЫ иметь возможность регистрировать свои StatEffect
- Custom эффекты ДОЛЖНЫ работать так же как встроенные

### FR-7: Commands

**FR-7.1: Stats Info Command**
- Команда `/bbf stats info [player]` ДОЛЖНА показывать:
  - Все характеристики с базовыми и итоговыми значениями
  - D&D модификаторы
  - Активные модификаторы по источникам
- Без аргумента ДОЛЖНА показывать статы текущего игрока
- С аргументом ДОЛЖНА показывать статы указанного игрока (требует OP)

**FR-7.2: Stats Set Command**
- Команда `/bbf stats set <stat> <value> [player]` ДОЛЖНА:
  - Устанавливать базовое значение характеристики
  - Требовать permission level >= 2
  - Применять изменения немедленно
  - Показывать подтверждение с новым значением

**FR-7.3: Stats Modifier Command**
- Команда `/bbf stats modifier add <stat> <source> <value> [player]` ДОЛЖНА добавлять модификатор
- Команда `/bbf stats modifier remove <source> [player]` ДОЛЖНА удалять все модификаторы от источника
- Команды ДОЛЖНЫ требовать permission level >= 2

**FR-7.4: Stats Reload Command**
- Команда `/bbf stats reload` ДОЛЖНА:
  - Перезагружать все конфиги персонажей и мобов
  - Переприменять эффекты ко всем сущностям
  - Требовать permission level >= 2
  - Показывать количество перезагруженных конфигов

### FR-8: Event Integration

**FR-8.1: Player Join Event**
- При входе игрока система ДОЛЖНА:
  - Загружать CharacterStatProfile из конфига
  - Создавать EntityStatData с базовыми значениями
  - Применять расовые модификаторы (если раса зарегистрирована)
  - Применять классовые модификаторы (если класс зарегистрирован)
  - Прикреплять данные к игроку через Attachment API
  - Применять все эффекты

**FR-8.2: Mob Spawn Event**
- При спавне моба система ДОЛЖНА:
  - Проверять наличие конфига для типа моба
  - Если конфиг есть - загружать и применять характеристики
  - Если конфига нет - использовать дефолтные значения или пропускать

## Non-Functional Requirements

### NFR-1: Performance

**NFR-1.1: Computation Efficiency**
- Пересчёт StatValue ДОЛЖЕН происходить только при изменении модификаторов
- Эффекты НЕ ДОЛЖНЫ применяться каждый тик
- Конфиги ДОЛЖНЫ кешироваться в памяти после загрузки

**NFR-1.2: Memory Usage**
- EntityStatData ДОЛЖЕН быть иммутабельным data class
- Модификаторы ДОЛЖНЫ храниться компактно (List, не Map)

### NFR-2: Reliability

**NFR-2.1: Error Handling**
- Отсутствие конфига персонажа НЕ ДОЛЖНО крашить сервер
- Невалидные значения в конфиге ДОЛЖНЫ автоматически исправляться
- Неизвестные statId в конфиге ДОЛЖНЫ игнорироваться с предупреждением

**NFR-2.2: Data Integrity**
- Значения характеристик ВСЕГДА ДОЛЖНЫ быть в допустимом диапазоне
- Модификаторы ДОЛЖНЫ корректно удаляться при удалении источника
- Attachment данные ДОЛЖНЫ корректно сохраняться и загружаться

### NFR-3: Extensibility

**NFR-3.1: API Design**
- StatRegistry ДОЛЖЕН быть публичным API
- StatEffect ДОЛЖЕН быть fun interface для лёгкой реализации
- Другие моды ДОЛЖНЫ иметь возможность добавлять характеристики и эффекты

**NFR-3.2: Backward Compatibility**
- Добавление новых характеристик НЕ ДОЛЖНО ломать существующие конфиги
- Удаление характеристики ДОЛЖНО игнорироваться в старых конфигах

### NFR-4: Security

**NFR-4.1: Permission Control**
- Все команды изменения характеристик ДОЛЖНЫ требовать OP (level >= 2)
- Игроки НЕ ДОЛЖНЫ иметь возможность изменять свои характеристики
- Конфиг-файлы ДОЛЖНЫ читаться только сервером

**NFR-4.2: Value Validation**
- Все входные значения ДОЛЖНЫ валидироваться
- Значения ДОЛЖНЫ зажиматься в [minValue, maxValue]
- Невозможно установить невалидные значения через команды или конфиги

### NFR-5: Maintainability

**NFR-5.1: Code Organization**
- Код ДОЛЖЕН следовать архитектуре из design.md
- Слои ДОЛЖНЫ быть чётко разделены:
  - api/ - публичные интерфейсы
  - component/ - data classes
  - registry/ - реестры
  - system/ - бизнес-логика
  - command/ - команды

**NFR-5.2: Documentation**
- Публичные API ДОЛЖНЫ иметь KDoc комментарии
- Сложные алгоритмы ДОЛЖНЫ иметь пояснения
- Формулы ДОЛЖНЫ быть задокументированы

## Acceptance Criteria

### AC-1: Basic Functionality
- [ ] Игрок с конфигом может зайти на сервер и получить характеристики
- [ ] Команда `/bbf stats info` показывает все 6 характеристик с корректными модификаторами
- [ ] Команда `/bbf stats set STR 18` изменяет силу и применяет эффекты
- [ ] Характеристики сохраняются после перезахода

### AC-2: Modifier System
- [ ] Расовые бонусы применяются при входе игрока
- [ ] Можно добавить модификатор через команду
- [ ] Можно удалить все модификаторы от источника
- [ ] Модификаторы корректно складываются

### AC-3: Configuration
- [ ] Конфиг персонажа загружается из JSON файла
- [ ] Невалидные значения зажимаются и логируются
- [ ] Команда `/bbf stats reload` перезагружает конфиги
- [ ] Отсутствие конфига не крашит сервер

### AC-4: Effects
- [ ] Изменение CON влияет на максимальное HP
- [ ] Изменение DEX влияет на скорость передвижения
- [ ] Эффекты применяются при изменении характеристик
- [ ] Старые эффекты удаляются перед применением новых

### AC-5: Extensibility
- [ ] Можно зарегистрировать новую характеристику через StatRegistry
- [ ] Можно создать custom StatEffect
- [ ] Custom характеристика работает так же как встроенные

## Out of Scope

Следующие функции НЕ входят в текущий спек:
- Интеграция с Origins API (расы) - будет в отдельном спеке
- Интеграция с системой классов - будет в отдельном спеке
- Навыки (skills) - отдельная система, зависит от характеристик
- UI для отображения характеристик - будет позже
- Система бросков костей - отдельный спек
- Автоматическое распределение очков при создании персонажа - через конфиги

## Dependencies

- Fabric API (Data Attachment API, Event API)
- Fabric Language Kotlin
- Mojang Serialization (Codec)
- Существующая система левелинга (PlayerLevelData)

## Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Конфликт с другими модами, изменяющими EntityAttribute | High | Medium | Использовать уникальные UUID для AttributeModifier, документировать |
| Производительность при большом количестве модификаторов | Medium | Low | Кешировать вычисленные StatValue, пересчитывать только при изменении |
| Сложность интеграции с Origins API | Medium | Medium | Сделать гибкий слой модификаторов, который легко подключить к Origins |
| Невалидные конфиги от администратора | Low | High | Автоматическая валидация и исправление, подробные логи |
